# Codex 详细攻略 知识点笔记

## 概览

* Codex 已成为当前热门的 AI 编程工具（超越 Cloud Code）。
* 优势：性价比高、对中文用户友好、相对稳定（较少遇到限速/封号问题）。
* 本笔记覆盖：四种运行环境、进阶功能、免费接入国产模型方法、实战对比与常见问题与解决办法。

## 二、命令行（CLI）—— 安装与基本流程

### 1. 前置：安装 Node.js

* 到 Node.js 官网下载并安装对应系统版本（Windows/macOS/Linux）。
* 安装后可通过 `node -v` / `npm -v` 校验。

**补充说明**：Node.js 是为了运行某些 Codex 客户端/SDK 与本地脚本；确保在 PATH 可用。

### 2. 安装 Codex CLI

* 在 Codex 的 GitHub 页面找到安装命令并在终端执行（字幕示意：复制命令到终端回车安装）。
* 安装后，在任意目录右键打开命令行，输入 `codex` 启动（将弹出 ChatGPT 登录页面，登录后回到终端继续交互）。

### 3. 登录与验证

* 启动 `codex` 后，页面会要求登录 ChatGPT（即你的 OpenAI/ChatGPT 账户）。
* 成功登录后，在 CLI 中输入测试消息，如 “hi”，应能收到响应。

## 三、模型与档位（在 CLI 中切换）

* 使用命令：`/model`（或 CLI 内对应的 model 切换命令）来选择模型。
* **模型类型举例（字幕出现）**：

  * `gpt-5`（通用、广泛知识）
  * `gpt-5-codex`（OpenAI 出品、专门优化编程/任务处理）
* **档位（质量/速度权衡）**：

  * `low`：响应最快，推理效果较弱（适合速率优先场景）
  * `medium`：折中（常用）
  * `high`：推理效果最好，但响应最慢（用于高质量输出）

**补充说明**：不同实现中档位命名可能有所不同（例如 `fast`/`balanced`/`quality`），以你当前 `codex` 版本的说明为准。

## 四、使用国产免费模型（以“ModelScope”为例）

**总体目标**：把第三方国产模型（提供推理 API）接入 Codex，实现免费或低成本使用。

### 步骤（基于字幕）

1. 在 `ModelScope` 社区注册账号（每天提供约 **2000 次免费推理调用**）。
2. 到「账号设置 → 访问令牌（API Key）」新建 Token，记下 `API Key` 与 `base_url`（推理 API 的 endpoint）。选中 `模型库` ，在筛选器中勾选 `推理 API- Inference`。选中模型点击进入卡片， 点击 `查看模型返范例`，主要需要  `base_url` 和 `api_key`.
3. 本地找到 Codex 配置文件夹（Windows 通常在 `C:\Users\<用户名>\codex\`；macOS/Linuxs 在界面提示的位置或类似 `~/codex`，不同版本可能略有差异）。
4. 编辑 `config.toml`（或 `CONFIG.TOML`）：

   * 删除原有内容或新建文件；
   * 填入模型名称和 `base_url`（但不要在文件中直接写明 API Key；用**环境变量**来存储密钥）。
5. 在搜索栏里搜索「环境变量」，点击「环境变量」按钮，在 Windows 系统的「环境变量」中新增变量（变量名为你在 `config.toml` 中引用的名字），把 `ModelScope` 社区给的 `API_KEY` 填入并保存。
6. 保存 config 文件，重启 `codex`，此时 `codex` 会用自定义模型直接请求 `ModelScope` 社区的推理接口。

### 示例（**config.toml** 的示意片段）

```toml
# 指定当前使用的模型名称
# 这里使用的是智谱AI（ZhipuAI）提供的 GLM-4.6 模型
model = "ZhipuAI/GLM-4.6"

# 指定模型提供方（Provider）
# 这里使用的是 Modelscope 平台（阿里达摩院提供的模型社区）
model_provider = "modelscope"

# 以下是针对特定 provider（modelscope）的详细配置块
[model_providers.modelscope]

# provider 名称（可以自定义，用于标识）
name = "Modelscope"

# 模型推理接口的基础 URL
# Codex 会将 API 请求发送到该地址
base_url = "https://api-inference.modelscope.cn/v1"

# 指定保存 API Key 的环境变量名称
# 注意：不要在此直接写入密钥，而是在系统环境变量中定义该键值
# 例如在 Windows 的“系统环境变量”中新建：
#   变量名：MODELSCOPE_API_KEY
#   变量值：你的实际 API Key
env_key = "MODELSCOPE_API_KEY"

```

**补充说明**：

- 不同社区/服务端返回的 API 格式不同，务必参考该服务提供的“推理 API 示例代码”来填写 `base_url` 和必要参数。

## 五、CLI 常用命令（重要命令与说明）

* `/init`

  * 功能：让 Codex 通读当前文件夹（项目），并生成一个 `agents.md`（或类似名）文件，记录 Codex 从项目中学到的上下文信息。
  * 作用：后续对话会把该文件内容作为上下文注入，提高理解与连贯性。
  * **补充**：该文件是通用格式（可被其他 AI 编程工具识别），可手动编辑以补充重要信息（如项目使用的技术栈：`Next.js 15` 等）。
* `/compact`
 
   * 功能：压缩对话上下文，剔除不重要内容，降低 token 消耗并提升模型专注度。
* `/new`

  * 功能：新建对话，会清空之前的对话历史；用于开始新任务或避免上下文污染。
* `/approvals`

  * 功能：调整 Codex 的自动执行权限，有三个档位：

    * `read-only`：所有修改需人工审核；
    * `auto`：仅敏感操作需人工审核；
    * `full-access`：无需人工审核，Codex 可自动执行修改（危险性与效率并存）。
* `/mcp`

  * 功能：列出已安装的 MCP（Modular/Managed/Marketplace CP 工具，MCP 是一种标准化工具箱），这些工具可被 AI 调用来操作外部工具或服务（如 Excel、文档检索、服务器等）。 

* **contex7项目**  是用来给 AI 提供一些最新的技术文档。

  **contex7项目:** [https://github.com/upstash/context7](https://github.com/upstash/context7)

## 六、MCP（第三方工具集）配置与示例

* **用途**：扩展 Codex 能力，使其能够通过标准化接口调用外部服务（例如：检索最新技术文档、操作 Excel、执行工具链等）。
* **安装流程（字幕示例）**：

  1. 在 MCP 的 GitHub 页面找到 Codex 的配置片段（示例为 `context7m cp server`）。
  2. 将配置加入 `codex` 的 `config.toml`（或 Config 文件）底部；不同 OS 可能需要不同的格式（字幕指出 Windows 需要特定格式）。
  3. 重启 Codex，执行 `/mcp`，应能看到刚安装的 MCP（如 `context7`）。
* **Excel MCP 示例**（字幕）：

  * 有一个 `excel mcp server`，允许 AI 生成并操作 Excel 数据（字幕演示：生成 20 个学生成绩并输出 Excel）。
  * 配置为 Python MCP 时，需指定运行命令（如 `uvicorn`）与 STDIO/参数等（字幕中示例需把 `-Y` 参数处理好，改为 `,` 形式等 Windows 特殊注意项）。

**补充说明**：MCP 的具体配置格式与字段依赖于 MCP 的实现；按该 MCP 的 README / 文档为准来填 `endpoint`、`command`、`STDIO` 等字段。

## 七、自定义命令（Prompts 文件）

* 在 Codex 的配置文件夹中新建 `prompts` 文件夹。
* 每创建一个 `.md` 文件就会生成一个自定义命令，文件名即为命令名。
* 支持占位符参数（例如：创建 `git_diff.md`，文件内定义两个占位符，运行 `/git_diff branch1 branch2` 即可）。
* **使用场景**：把常用复合提示封装成命令（例如代码审查模板、分支对比总结、标准化 PR 评论等）。

**示例**：`git_diff.md` 内可写：

```
# prompt template
对比分支$1 与 $2 的差异，并用自然语言输出要点摘要、潜在风险与建议。
```

## 八、常见 Bug 与解决方案

* **问题**：Codex 在修改文件时有时偏向使用 `shell` 工具去修改（效率低、容易出错）。
* **解决**：在对话或提示词中强制使用 `apply patch`（Codex 自带工具）来修改代码，或在中途打断并指示使用 `apply patch`。
* **补充建议**：把该偏好写入自定义命令或提示模板，降低复现概率。

## 九、IDE 插件（以 VS Code 为例）

### 功能亮点

* 安装官方 Codex 插件（OpenAI 出品）后，与 CLI 配置共享历史/命令。
* 插件支持：

  * **可视化文件对比**（直接弹出差异视图）；
  * **Undo（撤回）按钮**：可一键撤回由 AI 做的改动（多次撤回能回退多个步骤）；
  * **在对话窗口粘贴图片**：方便将截图/设计稿插入对话作为上下文；
  * 插件会保留 CLI 的历史和 config 配置，减低重复配置成本。
* 操作流程示例：在 VS Code 中选中按钮改色、点击插件生成的对比视图 → 审核 → 确认 / 撤回。

**补充说明**：VS Code 插件与 CLI 同步，插件的「撤回」功能对迭代式改动非常方便，适合快速尝试 AI 改动并回退。

## 十、云端运行（Codex 网页版 + GitHub 集成）

### 功能与流程（按字幕）

1. 在 ChatGPT/ Codex 网页端选择 “云端运行” 选项。
2. 将本地项目 **发布到 GitHub**（VS Code → `Publish to GitHub`，可选私有仓库）。
3. 在 Codex 网页端绑定 GitHub 账户，选择要操作的仓库。
4. 输入修改请求（例如：让按钮更显眼），Codex 会在云端自动创建运行环境并执行修改。
5. Codex 会在云端执行修改并生成变更日志，修改以 **Pull Request（PR）** 的形式提交到 GitHub。
6. 在网页上可以审阅 PR，满意后合并（或通过按钮直接合并到主干）。
7. 回到本地 VS Code 可以 `Sync` 同步远端改动到本地分支。

### 云端自动代码审核

* 可在 Codex 网页端开启「自动代码审核」设置。
* 启用后，每次 PR Codex 会自动生成 code review 建议（例如：指出将 HTTPS 改为 HTTP 的危险性），帮助提前发现 BUG 与不合理改动。

**补充说明**：云端运行使得在无本地环境的情况下也能完成修改、测试与 PR 流程；但需要注意权限与 PR 合并策略以防误合并到主分支。

## 十一、SDK（将 Codex 接入你的代码）

* 示例流程（字幕描述）：

  1. 新建文件夹，执行 `npm init -y` 把文件夹初始化为 Node 项目。
  2. 安装 Codex SDK（`安装 codex SDK`，按官网或 npm 名称安装）。
  3. 新建示例代码文件（示例把官网的示例代码复制粘贴过来），配置跳过 Git 安全检查（如示例所需）。
  4. 运行 `node <file>`，若输出正常即表示 SDK 配置成功，本地代码可通过 SDK 调用本地/已配置的 Codex 实例完成自动化任务。

**补充说明**：由于 SDK 包名可能随版本变化，实际安装请参考 Codex 官方文档或 GitHub repo 提供的安装指引；SDK 通常允许无须在代码中显式写入 API Key（如果本机上已有 Codex 登录会直接调用本地服务）。

## 十二、实战比较（你画我猜在线游戏）

* **任务**：创建一个「你画我猜」在线游戏（画布绘制 + AI 猜测）。
* **模型对比（字幕评分）**：

  * `GPT5`（通用模型）：一次性完成功能，页面美观，功能正常 → **97 / 100**。
  * `GPT5codex`（编程专用模型）：一次性完成功能，少量 UI 细节（按钮）可改进 → **95 / 100**。
  * `GLM4.6`（国产模型示例）：存在 bug（猜测结果显示黑屏，可能是画布背景未设置好）→ **80 / 100**。
* **结论**：

  * `GPT5` + Codex 表现最好，与 Cloud Code 持平；
  * Codex 的 Plus 会员额度通常对个人用户友好，不易触发速率上限；
  * 对中国用户体验友好，较少遇到 Cloud Code 那类封号或限速问题。

**补充说明**：模型表现是随版本与 prompt/上下文而变的，实测应以当前模型版本与你的 prompt 为准。

## 十三、权限与安全建议

* 对自动执行权限慎重配置：除非信任并完全测试，建议不要开启 `full-access`。
* 对接第三方模型时，不要在 config 文件中明文写入 API Key，使用环境变量存储并限制机器/账号权限。
* 在云端运行或授权 Codex 操作 GitHub 仓库前，建议使用受限的分支保护策略与 PR 审核流程。

## 十四、实用提示 & 常见操作速查表

* 初始化项目上下文：`/init` → 生成 `agents.md`（手动补充关键信息如技术栈）。
* 清理上下文降低 token：`/compact`。
* 开始新任务清空上下文：`/new`。
* 调整自动执行权限：`/approvals`（`read-only` / `auto` / `full-access`）。
* 列出 MCP：`/mcp`。
* 自定义命令：在 `prompts/` 目录新增 `.md` 文件（文件名即命令）。
* 遇到 Codex 偏向 shell 编辑导致出错：中断并强制使用 `apply patch`。
* 插件撤回改动：VS Code 插件面板中的 `undo`（可逐步回退 AI 改动）。
* 云端 PR 流程：发布到 GitHub → 绑定 GitHub → Codex 在云端修改并发 PR → 审核 & 合并 → 本地 `Sync`。

## 十五、重要资源与约束（要点）

* **使用限制**
    * ModelScope 推理 API-Inference，旨在为开发者提供免费的便捷模型调用方式，请勿用于需要高并发以及 SLA 保障的线上任务，如有商业化使用的需求，建议使用各商业化平台的 API。
    * 免费推理 API 由阿里云提供算力支持，要求您的 ModelScope 账号必须绑定阿里云账号后才能正常使用。
    每位 ModelScope 注册用户，当前每天允许进行总数为 2000 次的API-Inference 调用，其中每单个模型不超过 500 次，具体每个模型的限制可能随时动态调整。
    * 在上述调用次数限制的基础上，不同模型允许的调用并发，会根据平台的压力进行动态的速率限制调整，原则上以保障开发者单并发正常使用为目标。出于资源等因素考虑，在每个模型每天不超过 500 次调用的基础上，平台可能对于部分模型再进行单独的限制，这包括单个模型的单天调用总数限制，以及并发限制。
* Codex 的官方模型（OpenAI）需 ChatGPT Plus 订阅权限；但通过自定义模型可以绕开 API 付费，使用国产推理服务（受服务商配额限制）。
* Windows / macOS 在配置路径与 config 格式上可能存在差异（字幕中特别提示 Windows 需要特别格式）；按各自系统的 config 文件路径和格式配置。

## 十六、补充建议（实际使用时）

1. **记录 `agents.md` 的关键字段**：项目语言、框架版本、依赖项、运行脚本、测试命令、常见入口文件（便于 Codex 更好地理解项目）。
2. **为自定义模型写好 Prompt Template**，并放到 `prompts/` 里以便复用，减少在运行时频繁修改。
3. **为 MCP 与外部服务编写测试用例**，证明其行为安全且可回滚。
4. **开启分支保护与 PR 模板**，即便 Codex 自动生成 PR，也能保证审阅流程与合并策略。
5. **把常用的一键撤回命令写入 README**（告诉团队如何一键撤销 Codex 的改动）。
