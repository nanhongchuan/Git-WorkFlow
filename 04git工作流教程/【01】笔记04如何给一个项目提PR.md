# 如何给一个项目提PR

![alt text](image-2.png)

给一个项目提 PR（Pull Request，本质是“把你的修改请求合并进原项目”）其实就一套固定流程。你可以理解为：**复制一份 → 改 → 提交 → 请求合并**。

我用你熟悉的工程视角（Git + 实战）给你讲清楚👇

---

# 一、PR 的本质是什么？

PR（Pull Request）最常见在 GitHub 上，本质是：

👉 “我改好了代码，你们来 review 一下，要不要合进主分支？”

---

# 二、标准流程（最重要）

## Step 1：Fork 项目（复制一份）

在 GitHub 上点击 **Fork**

👉 相当于：

> 把别人的仓库复制到你自己的账号下

---

## Step 2：clone 到本地

```bash
git clone https://github.com/你的用户名/项目名.git
cd 项目名
```

---

## Step 3：添加上游仓库（很关键）

```bash
git remote add upstream 原项目地址
```

👉 以后你就可以同步原项目更新

---

## Step 4：新建分支（不要直接改 main！）

```bash
git checkout -b fix-bug-xxx
```

👉 命名建议：

* fix-xxx（修 bug）
* feat-xxx（新功能）
* docs-xxx（文档）

---

## Step 5：修改代码 + 提交

```bash
git add .
git commit -m "fix: 修复 xxx 问题"
```

---

## Step 6：推送到你的仓库

```bash
git push origin fix-bug-xxx
```

---

## Step 7：发起 PR（核心一步）

去 GitHub 页面，会自动提示：

👉 **“Compare & pull request”**

填写：

* 标题（你做了什么）
* 描述（为什么这么改）

点击提交 ✅

## 一般去 **你自己 fork 的项目页面** 或 **原项目页面** 都行，GitHub 都可能提示。

但最稳的做法是：

**去你自己 fork 的项目页面**
也就是：

```text
https://github.com/你的用户名/项目名
```

你 push 分支后，页面上通常会出现：

```text
Compare & pull request
```

点进去后，确认方向是：

```text
base repository: 原项目
base branch: main

head repository: 你的 fork
compare branch: 你的分支
```

也就是：

```text
你的 fork 分支  →  原项目 main 分支
```

举例：

```text
your-name/project:fix-bug
        ↓
original-owner/project:main
```

千万别反了。

你可以这样记：

**代码从你这里出去，所以 head 是你的 fork；代码要合到原项目，所以 base 是原项目。**

---

# 三、PR 的完整链路（你可以这样理解）

👉 一句话版：

> Fork → 改代码 → Push → PR → 被 review → 合并

👉 类比现实：

* Fork = 复制一份公司代码
* 分支 = 你自己开个实验室
* PR = 提交变更申请
* Maintainer = 项目负责人审批

---

# 四、高手会做的几个细节（很关键）

## 1. 先同步 upstream（避免冲突）

```bash
git fetch upstream
git merge upstream/main
```

---

## 2. PR 要“小而清晰”

不要一口气改一堆：

❌ 一个 PR 改 1000 行
✅ 一个 PR 只干一件事

---

## 3. 写清楚 PR 描述

好的 PR 描述：

* 问题是什么
* 你怎么解决的
* 有没有影响其他模块

---

## 4. 遵守项目规范

每个项目可能有：

* lint 规则
* commit 规范（如 `feat:`、`fix:`）
* PR 模板

👉 不遵守直接被 reject

---

# 五、常见坑（你大概率会踩）

## ❌ 1. 直接在 main 上改

👉 会被拒

---

## ❌ 2. 没有同步 upstream

👉 容易冲突

---

## ❌ 3. 提交太乱

```bash
update code
fix something
test
```

👉 这种 commit 会被嫌弃

---

## ❌ 4. PR 太大

👉 review 不过来，直接拖死

---

# 六、你可以这样记忆（最简模型）

👉 PR = **代码投简历**

* 你写代码 = 写作品
* PR = 投递
* Maintainer = 面试官
* Review = 面试过程
* Merge = 入职成功
